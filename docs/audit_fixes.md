# Codebase Audit & Security Fixes

This document details the security and stability improvements made during the codebase audit on `printease-mvp-main`. 

## 1. Hub Pricing Update Stability
**Files Modified**: 
- `backend/src/db/repository.js`
- `backend/src/controllers/centreController.js`

**What changed**:
1. Added validation logic in `centreController.js` (`updateCentrePricing` method) to ensure that `bwSingle`, `bwDouble`, `colorSingle`, `colorDouble`, and `watermarkCharge` are strictly numbers between 0 and 10000.
2. If invalid values are provided, the API now safely returns a `400 Bad Request` instead of propagating the error to the database.
3. Updated the `updateCentrePricing` repository function to map `undefined` values to `null`. 

**Why it changed**:
When a Hub owner updated their pricing, missing fields in the JSON payload evaluated to `undefined`. Passing `undefined` directly into the PostgreSQL (`pg`) driver caused an unhandled constraint error resulting in a `500 Internal Server Error`. Replacing `undefined` with `null` allows the database's `coalesce` function to correctly keep the old values without crashing the node process.

## 2. Order Status Transitions Security
**Files Modified**: 
- `backend/src/controllers/orderController.js`

**What changed**:
Modified the `updateOrderStatus` method to fetch the existing order *before* saving the new status. We added validation to prevent changing the status of an order if it is currently marked as `Collected` or `Cancelled` (unless the target status is the exact same status).

**Why it changed**:
Previously, a hub owner could inadvertently (or maliciously) transition a completed or cancelled order back to a working state (like "Draft" or "Printing"). Validating state transitions strictly prevents data corruption and prevents print agents from re-processing dead orders.

## 3. Centre Lookup Rate Limiting
**Files Modified**: 
- `backend/src/middleware/rateLimitMiddleware.js`
- `backend/src/routes/centreRoutes.js`

**What changed**:
Created a new `centreLookupRateLimit` (max 30 requests per minute by default) and attached it to the `GET /:code` endpoint in `centreRoutes.js`.

**Why it changed**:
The public `GET /api/centres/:code` endpoint used to resolve a print hub's code had no rate limits. An attacker could brute-force guess hub codes or spam the database. The new rate limit mitigates DoS risks on the `findCentreByCode` database query.

## 4. Desktop App Auto-Updater Verification
**Files Audited**: 
- `printease-desk/desktop-shell/updater.js`

**Findings**:
The desktop app correctly disables `updater.autoDownload = false`. However, it triggers `updater.downloadUpdate()` manually when an update event is captured. After downloading, it prompts the user using a dialog (`"Update downloaded. Install now?"`). This is fully functional and provides a proper manual UX. No code changes were required as the flow is safe and transparent to users.

## 5. Webhook & Download Link Validation
**Files Audited**:
- `paymentController.js`
- `documentController.js`
- `agentController.js`

**Findings**:
1. **Razorpay Webhooks**: Validation function correctly short-circuits on mismatched payload signature lengths before evaluating `crypto.timingSafeEqual`, preventing NodeJS type errors and DoS crashing.
2. **Document Ownership**: Private document storage strictly enforces ownership checks via `findDocumentAccessContext`, joining across `print_order_files` and `print_orders` to ensure users and hub agents can only download related files. TTL is correctly capped.
3. **Agent Registration**: Generating print agent tokens requires scanning a signed URL generated by an authenticated hub dashboard, effectively bounding rogue device registrations.

All high-risk areas evaluated as completely secure.
